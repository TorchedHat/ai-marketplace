## When to Graph Break vs Fix

**Graph break is acceptable when**:
- Feature is rarely used
- Implementation is very complex
- Dynamic behavior is inherent

**Implement support when**:
- Feature is commonly used
- Performance matters
- Implementation is straightforward

## Compilation Mode Context

Dynamo behaves differently depending on compilation mode:

- **`torch.compile`**: `tx.export=False`, no `_compiling_state_context()` active. Standard Dynamo+Inductor flow. Graph breaks are allowed.
- **`torch.export` (strict)**: `tx.export=True`, `_compiling_state_context()` active. Stricter tracing - no graph breaks, stricter guard semantics.
- **`torch.export` (non-strict, **the default**)**: Default export mode, uses `fullgraph_capture`. Check `torch._export.config.use_new_tracer_experimental` for tracer variant.

**Why this matters for implementation**: Export mode has different requirements (no graph breaks allowed, different guard generation). When implementing VariableTracker methods or opcode handlers, check `tx.export` to handle mode-specific behavior.

**Implementation pattern**:
```python
# In VariableTracker method or opcode handler
if tx.export:
    # Export mode - stricter requirements
    # Cannot use unimplemented() for graph breaks
    # Must fully support the operation
    result = self._fully_implement_operation(tx, args, kwargs)
else:
    # torch.compile mode - graph breaks OK if needed
    if not self._can_handle_operation():
        unimplemented("operation_name", "reason for graph break")
    result = self._implement_operation(tx, args, kwargs)
```

## Quick Start

**Have a compilation error?**

1. Create minimal repro
2. Enable logging
3. Follow the 4-category workflow below

## Minimal Reproduction

Always start with the smallest possible repro:

```python
import torch

def fn(x):
    return x + 1  # Simplify to just the failing operation

x = torch.randn(4)
compiled_fn = torch.compile(backend="eager")(fn)
compiled_fn(x)  # Should trigger the error
```

## Enable Debug Logging

```python
import torch
import logging

# Enable Dynamo logging
torch._logging.set_logs(dynamo=logging.INFO)

# Also enable bytecode logging
torch._logging.set_logs(dynamo=logging.INFO, bytecode=True)

# For guard failures
torch._dynamo.config.verbose = True
```

## Advanced Debugging Tools

### comptime.breakpoint()

Drop into pdb **during compilation** to inspect Dynamo's internal state:

```python
import torch._dynamo.comptime as comptime

def fn(x):
    comptime.breakpoint()  # Drops into pdb during tracing
    return x + 1

torch.compile(fn)(torch.randn(4))

# In pdb session, use ComptimeContext (ctx):
# ctx.print_locals()  - See all VariableTrackers in scope
# ctx.print_bt()      - See bytecode execution stack
# ctx.print_graph()   - See FX graph being built
# ctx.get_local("x").as_fake()  - Inspect specific variable
```

**When to use**: Investigating VariableTracker state, understanding symbolic execution flow, debugging complex bytecode sequences.

### BytecodeDebugger

Step through **generated bytecode** to debug reconstruction errors and runtime crashes:

```python
import torch._dynamo.bytecode_debugger as bd

with bd.debug():
    compiled_fn(x)

# Commands:
# v (verbose) - Print every instruction before execution
# c (continue) - Run until completion or exception
# Last printed instruction before crash = culprit
```

**When to use**: Debugging segfaults (no Python traceback), reconstruction errors, codegen bugs.

**Programmatic breakpoints**:
```python
# In user code - auto-activates debugger
torch._dynamo.bytecode_debugger.breakpoint()

# Or in Dynamo codegen (torch/_dynamo/codegen.py)
codegen.extend_output(create_breakpoint())
```

## 4-Category Debugging Workflow

Check these in order:

### 1. Bytecode Capture Incorrect

**Symptoms**:
- Wrong instructions being traced
- Missing operations
- Incorrect control flow

**Debug**:
```python
torch._logging.set_logs(bytecode=True)
# Look at "ORIGINAL BYTECODE" vs "MODIFIED BYTECODE"
```

**Check**:
- Is the right function being compiled?
- Are all instructions present?
- Is the bytecode structure correct?

### 2. Object → Variable Conversion Incorrect

**Symptoms**:
- TypeError about VariableTracker
- Wrong type being used
- Unexpected behavior with custom objects

**Debug**:
Add logging in `variables/builder.py`:
```python
def __call__(self, value):
    print(f"Converting {type(value)} to VariableTracker")
    # ... rest of logic
```

**Check**:
- Is the Python object being converted to the right VariableTracker type?
- Are the conversion heuristics correct?
- Is `is_namedtuple()`, `isinstance()`, etc. working correctly?

**Common issues**:
- Namedtuples not recognized → Check `is_namedtuple()`
- Custom class treated as constant → Add to VariableBuilder
- Wrong VariableTracker type → Check isinstance() order

### 3. Variable Implementation Incorrect

**Symptoms**:
- AttributeError on VariableTracker
- Method calls failing
- Incorrect operations

**Debug**:
Check the VariableTracker class for the type:
```python
# If working with MyType
# Check variables/user_defined.py or create custom class
```

**Check**:
- Does `call_method()` handle the method?
- Does `call_function()` work correctly?
- Does `var_getattr()` handle attributes?
- Is `reconstruct()` implemented?

**Common issues**:
- Missing method handler → Add to `call_method()`
- Wrong return type → Return appropriate VariableTracker
- Not calling super() → May miss base class logic

### 4. Output Bytecode Incorrect

**Symptoms**:
- Runtime errors after compilation
- Wrong results
- Reconstruction failures

**Debug**:
Check `reconstruct()` method:
```python
def reconstruct(self, codegen):
    # Add logging
    print(f"Reconstructing {self}")
    # Check generated bytecode
```

**Check**:
- Is `reconstruct()` generating correct bytecode?
- Are side effects tracked?
- Is bytecode valid Python?

**Common issues**:
- Missing imports → Use `codegen.load_import_from()`
- Wrong argument order → Check `create_call_function()` args
- Side effects not tracked → Use `tx.output.side_effects`

## Guard Failures

**Symptom**: Frequent recompilation, performance issues

**Debug**:
```python
torch._dynamo.config.verbose = True
# Also:
import os
os.environ['TORCHDYNAMO_REPORT_GUARD_FAILURES'] = '1'
```

**Check output for**:
- Which guard is failing
- Why it's failing (value changed)
- How often it's failing

**Common causes**:
- Dynamic tensor shapes → Use dynamic shapes or fix size
- Changing types → Ensure consistent types
- Data-dependent control flow → May need graph breaks

**Solutions**:
- Make inputs consistent
- Use `torch._dynamo.config.assume_static_by_default = False` for dynamic shapes
- Accept graph breaks for dynamic behavior

## Common Error Patterns

### "Graph break" on simple operation

**Cause**: Operation not supported or improperly implemented.

**Check**:
1. Is there an opcode handler in `symbolic_convert.py`?
2. Does the VariableTracker have the method?
3. Is there a specific implementation needed?

**Fix**: Add opcode handler or method implementation.

### Slow compilation

**Causes**:
1. Too many guards → Recompiling frequently
2. Complex graph → Large model or operation
3. Tracing through Python → Missing fast-path

**Debug**:
```python
import torch._dynamo.utils as utils
utils.compile_times()  # See compilation times
```

**Solutions**:
1. Reduce dynamic behavior
2. Use smaller model for testing
3. Implement fast-path for operation

## Step-by-Step Debug Session

### Example: Debugging namedtuple issue

**Step 1**: Create repro
```python
import torch
import collections

MyTuple = collections.namedtuple("MyTuple", ["a", "b"])

@torch.compile
def fn(x):
    return MyTuple(x, x + 1)

fn(torch.randn(4))  # Error here
```

**Step 2**: Enable logging
```python
torch._logging.set_logs(dynamo=logging.INFO, bytecode=True)
```

**Step 3**: Check bytecode
- Is `MyTuple` being called correctly?
- Is the bytecode capturing the namedtuple creation?

**Step 4**: Check Variable conversion
- Add logging to `VariableBuilder`
- Is `MyTuple` being converted to the right VariableTracker?
- Should be `NamedTupleVariable`, not `UserDefinedObjectVariable`

**Step 5**: Check VariableTracker
- Does `NamedTupleVariable` exist?
- Does it have `call_function()` or proper initialization?
- Check `variables/lists.py`

**Step 6**: Check reconstruction
- Does `NamedTupleVariable.reconstruct()` work?
- Does it generate correct bytecode?

**Step 7**: Fix and test
- Implement missing logic
- Run test again
- Verify with full test suite


```python
from torch._dynamo import compiled_autograd
compiled_autograd.reset()
```

```python
import torch._dynamo.export
```

## Testing Your Fix

### Write a test

```python
# In test/dynamo/test_misc.py
class TestMyFix(torch._dynamo.test_case.TestCase):
    def test_my_case(self):
        def fn(x):
            # Your repro case
            return x + 1

        x = torch.randn(4)
        ref = fn(x)
        opt_fn = torch.compile(fn, backend="eager")
        res = opt_fn(x)
        self.assertEqual(ref, res)
```

### Run the test

```bash
cd pytorch
python test/dynamo/test_misc.py TestMyFix.test_my_case -v
```

### Run related tests

```bash
# Run all tests in the file
python test/dynamo/test_misc.py -v

# Run all dynamo tests (slower)
pytest test/dynamo/ -v
```

## Pro Tips

1. **Start simple**: Minimal repro is key
2. **Use eager backend**: For testing tracing without compilation
3. **Compare bytecode**: Original vs. reconstructed
4. **Check existing code**: Similar features are likely implemented
5. **Test incrementally**: Don't wait until everything is done
